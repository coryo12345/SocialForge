## What is SocialForge

A self-hosted, AI-generated social media simulator (Reddit-style clone). Every user, post, and comment is AI-generated via Ollama (llama3.1:8b). Content is pre-generated by Python scripts and revealed gradually via `scheduled_at` timestamps.

## Commands

### Client (port 5173)
```bash
cd client && npm run dev       # Start Vite dev server
cd client && npm run build     # tsc -b && vite build
```

### Server (port 3001)
```bash
cd server && npm run dev       # tsx watch index.ts
cd server && npm run build     # tsc
cd server && npm start         # node dist/index.js (production)
```

### Python content generation scripts
```bash
cd scripts && pip install -r requirements.txt
python generate_daily.py       # Run full daily content generation
python generate_posts.py       # Generate posts only
python generate_comments.py    # Generate comments only
```

## Architecture

### Directory layout
```
client/    React 19 + Vite + TailwindCSS v4 PWA (port 5173)
server/    Express 5 + TypeScript REST API (port 3001)
shared/    Type definitions shared by both (shared/types.ts)
scripts/   Python scripts that call Ollama and POST to /api/internal/*
data/      SQLite database (gitignored)
specs/     Phase design specifications
```

### Data flow
1. Python scripts call Ollama → generate content → bulk-insert via `/api/internal/*` (protected by `INTERNAL_API_KEY`)
2. Content has `scheduled_at` timestamps; the feed API filters `scheduled_at <= now()` to reveal it gradually
3. `server/jobs/scoreUpdater.ts` runs every 15 minutes to recalculate post/comment scores
4. React client fetches paginated data from `/api/feed` with cursor-based pagination

### Server structure
- `server/index.ts` — entry point, route mounting, error handling
- `server/db.ts` — SQLite init, WAL mode, FK constraints, migrations runner
- `server/migrator.ts` — umzug migration orchestrator
- `server/routes/` — one file per API resource (feed, posts, comments, communities, users, search, settings, activity, internal, auth)
- `server/middleware/session.ts` — cookie-based session; attaches `req.user`
- `server/migrations/` — numbered SQL migration files

### Client structure
- `client/src/App.tsx` — React Router v7 routes + theme provider
- `client/src/store/useSession.ts` — Zustand store (current user + theme)
- `client/src/api/client.ts` — Axios instance with `baseURL=/api`
- `client/src/pages/` — one file per route
- `client/src/components/` — shared UI components
- `client/src/styles/globals.css` — Tailwind v4 imports + CSS theme variables

### Shared types
`shared/types.ts` is imported by both client and server. The client reaches it via the path alias `shared/types` → `../shared/types.ts` configured in `client/tsconfig.json` and `client/vite.config.ts`.

### Authentication
Username-only login (no passwords). Sessions are stored server-side as cookies. The `/api/internal/*` endpoints use a separate `INTERNAL_API_KEY` header for the Python content pipeline.

## Environment variables

**server/.env** (copy from `server/.env.example`):
- `PORT=3001`
- `DB_PATH=../data/SocialForge.db`
- `INTERNAL_API_KEY` — used by Python scripts to call `/api/internal/*`
- `SESSION_SECRET` — cookie signing
- `CORS_ORIGIN=http://localhost:5173`

**client/.env** (copy from `client/.env.example`):
- `VITE_API_URL=http://localhost:3001` — only needed if not using the Vite proxy

In dev the Vite proxy forwards `/api/*` to `http://localhost:3001`, so both servers must be running simultaneously.

## Key tech choices
- **better-sqlite3** — synchronous SQLite; no async/await in DB calls
- **Express 5** — async route handlers throw errors automatically (no `next(err)` needed)
- **TailwindCSS v4** — uses `@tailwindcss/vite` plugin, not PostCSS
- **React Query v5** — all server state; Zustand only for session/theme
- **React Router v7 library mode** — not framework mode; routing is client-side only
- **umzug** — migration system; migrations in `server/migrations/`
